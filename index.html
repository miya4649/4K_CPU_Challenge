CPU Design Competition "4K CPU Challenge"

About 4K CPU Challenge

The 4K CPU Challenge is an open source CPU design competition.

Participants compete to see how far they can go in designing a practical CPU with less than 4KB of source code.

The duration of the competition is indefinite, and anyone can enter for free at any time.

Purpose of the competition

The purpose of this competition is to allow a large number of CPU developers to compete for the best CPU designs and to share their ideas and techniques with each other and with society as a whole by publishing the results as open source, thereby creating the ground for future CPU excellence.

Competition entry eequirements

Upload the source code of your own CPU design or open source ISA CPU (e.g. RISC-V) to your repository on GitHub and notify its URL in Issues.

  • Please keep the total bytes of the CPU core source code within 4KB.
  • Please write the CPU in a language such as HDL, HLS, DSL, etc. that can be implemented in hardware by FPGA synthesis tools or EDA tools.

The CPU core here is assumed to include the following functions at a minimum. However, it is not necessary to have them as hardware functions if they can be realized on the software side.

  • Function to execute arbitrary programs
  • Ability to perform numerical operations, bit operations, shift operations, and numerical comparisons
  • Must support control structures (loops, jumps, calls, returns, conditional branches)
  • ALU and register files should be included in the CPU core source code.
  • Instruction memory, data memory, bus and device I/O source code need not be included in the CPU core (4KB). (However, please submit the source code for the entire executable project.)
  • Minifying the CPU core source code (shortening variable names, etc.) in order to keep it within 4KB is acceptable. If you minify the source code, please also submit the original source code before minifying.
  • You must check the operation on at least one type of FPGA board and prepare a project file.
  • Prepare a simulator testbench project so that anyone can test it.
  • Prepare a software development environment with at least assembler support so that anyone can program.

Competition evaluation axis

We do not plan to award prizes at this time, but in the future we will consider ranking and presenting awards based on the following evaluation axes.

  • High speed of actual program execution (Integer Operations/sec, etc.)
  • Small circuit size (LUTs, etc.) when implemented
  • Short source code
  • High maximum operating frequency (fMax)
